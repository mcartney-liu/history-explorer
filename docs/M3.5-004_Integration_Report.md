## 1. 消费者定位与设计原则（UI 设计说明）

> 依据 `docs/M3.5-000_Schema_Freeze_Review.md` §1（消费者）与 §5.2（四类输出）。

History Explorer 的消费者是**历史爱好者 / 学生 / 内容创作者**——他们点开一个文明 / 事件 / 人物，想知道"它和什么有关、怎么影响到别的文明、后来怎样"。价值主张（M3.5-000 §1.3）：*把互联的历史知识图谱变成可解释的发现路径，让学习者追随思想/技术/事件跨越文明与时间，而不靠算法猜测。*

M3.5-004 据此把探索页设计为**五个叙事区**，逐一对应消费者的 Jobs-to-be-done（JT-1~JT-4）：

| 探索区 | 对应 JT | 消费者价值 |
|--------|---------|-----------|
| **Related Connections** | JT-1 | 即时邻居，跨主题可点，看懂"和谁相连" |
| **Explained Connections** | JT-4 | 每条连接带"为什么"的可解释说明 + score，建立信任 |
| **Exploration Paths** | JT-3 | 多跳"经由 X"路径链，降低探索门槛 |
| **Timeline** | JT-1/2 | 时间感知，呈现先后与共存 |
| **Themes** | JT-2 | 按关系/实体类型派生的跨文明线索（征服/继承/传播/贸易…） |

**明确不是什么**：不做后台管理界面、不做调试页面、不做 Graph Database Viewer（无 D3/Neo4j/图引擎，纯 CSS 芯片）。视觉沿用既有 parchment/bronze 主题（CSS 变量 `--he-accent` 等），无新增颜色 token。

## 2. 五大探索区设计（落实位置与渲染）

两个视图（实体页 `EntityPage.tsx` + 探索页 `App.tsx` 探索块）均按 1→5 顺序渲染五区：

1. **Related Connections** — 复用 `RelationshipView` + `RelatedEntityList`（渲染 `exploration.related_entities`）。本地 id 芯片走既有 `onEntityClick`（自动加 `topic:` 前缀），跨主题可点。
2. **Explained Connections** — `ConnectionsExplainedPanel` 渲染 `connections_explained[]`：每条卡片显示本地名 + `depth N` 青铜 pill + `score N` pill + italic bronze 解释（`explanation` 缺失回退 `global_id`）。
3. **Exploration Paths**（新增）— `ExplorationPathsPanel` 把每项的 `path`（`global_id[]`）渲染成左→右节点芯片链，节点间用关系边连接（`—[conquered outgoing]→`）。每芯片携带完整 `global_id`，经 `onNodeClick` 跨主题可点。把"罗马 → 丝路 → 汉"这类多跳叙事可视化。
4. **Timeline** — 复用 `TimelinePanel`（时间感知，未改）。
5. **Themes**（新增）— `ThemesPanel` **纯前端派生**跨文明线索：按关系类型分组为人类主题（`conquered`→Imperial Conquest、`inherited`→Cultural Inheritance、`spread`→Diffusion & Spread、`traded_with`→Trade & Exchange、`influenced`→Influence、`invented`/`discovered`→Innovation、`practiced`/`spoke`→Practice & Language，未知类型 titleCase）；并额外按冻结 `ENTITY_TYPES` 派生类型线索（Civilization/Technology/Religion/Idea）。成员芯片经 `other.global_id`/`other.topic` 跨主题可点。

**跨主题点击防重复前缀**：`onEntityClick(id)`（本地 id，加 `topic:` 前缀）仅用于 Related/Timeline 的本地芯片；`onNodeClick(globalId)`（已带完整 `global_id`）直接 `openEntity(globalId)`，不重复加前缀。两条路径在 `App.tsx:393-394 / :399-400 / :411-412` 严格分离，验证后端 `entity()` 能解析 `namespace:id`。
