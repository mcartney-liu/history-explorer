## 0. Scope & Constraints Honored

This checkpoint implements the **deterministic, explainable Exploration Engine** on
top of the M3.5-001 `GlobalGraph`. It answers the explorer's question:
*"In the interconnected historical world, why are these two entities related, and
what connects them?"*

Hard constraints (from M3.5-000 freeze + M3.5-002 brief), all honored:

| Constraint | Honored? | How |
|---|---|---|
| Don't modify `ENTITY_TYPES` / `RELATIONSHIP_TYPES` | ✅ | Schema enums untouched; the engine reads them via a **static interpretation map** (not the schema itself). |
| Don't modify `global_id` rules | ✅ | Reuses `GlobalGraph` canonical keys verbatim. |
| No AI / GIS / Neo4j / third-party graph lib | ✅ | Pure stdlib; every score is a fixed, explainable formula. |
| Not a Search engine | ✅ | Operates on graph structure + curated time fields, never free-text matching. |
| Not a Recommender | ✅ | Models "why two entities connect", not "what the user likes". |
| Not the shortest-path = best | ✅ | Enumerates candidate paths and **scores** them (relationship meaning leads). |
| Explainable output | ✅ | Every path keeps per-step relation type + direction; `explanation` + `score_breakdown` seams preserved. |
| No new API | ✅ | Only additive library methods on `KnowledgeService`. |
| No frontend change | ✅ | Frontend untouched. |

Explicitly **out of scope** (deferred to M3.5-003 / 004): API exposure, UI, recommendation.

## 1. Architecture — Where the Engine Lives

```
REST API (frozen)
   │  delegates
   ▼
KnowledgeService (Facade — aggregates, never implements algorithms)
   │
   ├── registry / graph / search / timeline / exploration   (M3-001)
   ├── GlobalGraph            (M3.5-001)  ← pure unified adjacency
   │      │  consumed by
   │      ▼
   └── ExplorationEngine      (M3.5-002)  ← deterministic scoring + explainable ranking
```

- **The engine is a Knowledge Layer module** (`core/exploration_engine.py`), NOT a
  standalone Service — consistent with M3-001's "no God Service" rule and M3.5-000's
  freeze. `KnowledgeService` owns one line: it *constructs* the engine at startup and
  *delegates* to it (no graph algorithm in the facade).
- **It consumes `GlobalGraph`** as its read-model: unified nodes (keyed by
  `global_id`) + cross-topic edges already included. The graph stays pure structure;
  ranking lives only here.
- **Construction inputs**: `GlobalGraph` + `KnowledgeRegistry` +
  `topic_datasets`. From these it builds two indexes **once**:
  - `_rel_lookup`: `(src_gid, tgt_gid, type) -> raw relationship dict` — so an
    explicit `weight` field (schema-reserved, currently null) overrides the static map.
  - `_importance`: per-entity importance from type salience + graph centrality.

### KnowledgeService wiring (additive, library-only)

```python
self._exploration_engine = ExplorationEngine(self._global_graph, self._registry, self._topic_datasets)

def get_exploration_engine(self) -> ExplorationEngine: ...
def explore_connections(self, src_gid, tgt_gid, max_depth=4, max_paths=16) -> dict: ...
def explore_from(self, gid, max_depth=2, limit=20) -> list[dict]: ...
```

These return **JSON-safe dicts** (dataclasses carry `to_dict()`), forming the future
M3.5-003 API/UI seam. No REST endpoint added → public API contract frozen.

## 2. Algorithm

### 2.1 Path discovery (not naive shortest-path)

- **Undirected, simple-path enumeration** over the `GlobalGraph` (relationship
  direction is *recorded* per step but does not block traversal — a `located_at`
  edge is still a valid connector when explored in reverse).
- Bounded by `max_depth` (default 4) and `max_paths` (default 16) → deterministic,
  bounded work even on larger graphs.
- For `explore_from`, reachability uses `GlobalGraph.subgraph(..., max_depth)` +
  `find_path` to get each node's best (shortest) connecting path.
- `find_path` itself reuses `DirectedGraph.find_path` (BFS) — no re-implemented traversal.

### 2.2 Deterministic scoring (4 explainable components)

For each candidate path, score =  
`0.35·relationship_meaning + 0.25·temporal_coherence + 0.20·entity_importance + 0.20·path_simplicity`

| Component | Meaning | Source |
|---|---|---|
| **relationship_meaning** | How story-relevant the link is (caused/influenced/ruled > located_at/related_to) | Static `RELATIONSHIP_MEANING` map (15 frozen types); explicit `rel.weight` overrides if present. |
| **temporal_coherence** | 1.0 if contemporary, decays with time gap (half-life 500y) | `start_date/end_date.value` (negative = BCE) on entity dicts. |
| **entity_importance** | Narrative centrality | `0.6·type_salience + 0.4·centrality` (centrality = `1 − 1/(1+degree)`, bounded). |
| **path_simplicity** | Rewards fewer hops w/ diminishing returns (`1/(1+0.4·(hops−1))`) | So 1-hop strong > 2-hop weak, but a 2-hop *strong* story can beat a 3-hop *weak* one. |

Weights are **centralized constants** at the top of the module — curated, not learned,
not ML. Relationship meaning leads; simplicity is the smallest term by design, so we
**never reward "shortest == best"**.

### 2.3 Time factor

- Representative year extracted as `start_date.value`, fallback `end_date.value`.
- `temporal_coherence(a, b)`:
  - both unknown → `0.5` (neutral)
  - one unknown → `0.6` (slight positive)
  - both known → `1 / (1 + |ya−yb| / 500)`
- BCE/CE handled uniformly (negative years), so "Rome (−27) ↔ Egypt (−3100)" scores
  lower than "Rome ↔ Han (−202)".

### 2.4 Relationship-type weight

15 frozen types mapped to meaning weights (see `RELATIONSHIP_MEANING`). If a
relationship ever carries a non-null `weight` (schema-reserved), that explicit value
wins — future-proofing without any schema change.

### 2.5 Explainability seam

Each `PathCandidate` / `ExploredNode` carries:
- `steps`: list of `PathStep{from_global_id, to_global_id, relationship, direction, weight}` —
  **relation type + direction preserved per node**.
- `score_breakdown`: the four components (auditable, debuggable).
- `explanation`: human-readable text, e.g.
  `"Civilization 'Rome' —[traded_with outgoing]→ Location 'Silk Road' —[traded_with outgoing]→ Civilization 'Han China'."`

This is the exact payload M3.5-003 (API) and M3.5-004 (UI) will surface.

## 4. Test Results

| Suite | Result |
|---|---|
| backend pytest | **94 passed** (85 baseline + 9 new `test_exploration_engine.py`) |
| frontend vitest | **38 passed** (frontend untouched) |
| `npm run build` | 51 modules, 0 errors |
| `test_api_v1` (8) | passed → API contract zero break |

`test_exploration_engine.py` covers (per brief):
1. **Single-hop** — `explore_from(rome, depth=1)` returns Silk Road with `traded_with` / `outgoing` preserved.
2. **Multi-hop** — Rome → Silk Road → Han China reachable at depth 2; path keeps intermediate node.
3. **Cross-topic** — connecting path spans `roman_empire` + `silk_road` topics (M3.5-001 property intact at engine layer).
4. **Time ordering** — synthetic equal-relationship edges: contemporary neighbor ranks above distant one; formula unit-tested (same year→1.0, 500y gap→0.5, unknown→neutral).
5. **Relationship meaning > naive shortest** — synthetic equal-length 2-hop paths: `influenced`×2 outranks `located_at`×2 (proves "shortest ≠ best").
6. **No path** — `explore_connections(rome, "no_such:node")` → empty paths + `"No connection"` explanation.
7. **Determinism** — `explore_connections` / `explore_from` produce byte-identical JSON across repeated calls.
