## 1. Executive Summary

M9-001 在冻结架构边界内，于现有确定性 `ExplorationEngine` 之上新增一层 **Deterministic Next-Node Recommendation Engine**：给定当前实体的 `global_id`，产出一组**可解释、可测试、可复现**的"下一站"推荐（默认 top 5）。

| 维度 | 结论 |
|---|---|
| 性质 | additive 推荐层；确定性图排序，**非 AI/ML/LLM runtime** |
| 后端范围 | **改 3 个文件**：`exploration_engine.py`、`knowledge_service.py`、`main.py` |
| 前端范围 | **M9-001 内不改前端**（遵循 M9-001 §4 Non-Goals / §11，前端消费推迟至 M9-002） |
| 数据范围 | 仅复用现有图 / JSON / 冻结枚举；**无 schema 变更、无 DB、无新依赖** |
| 新增端点 | `GET /entity/{entity_id}/recommendations`（方案 A，v1 + legacy 双挂载，向后兼容） |
| 测试 | 新增 `backend/tests/test_recommend.py`（确定性 / 排序 / 可解释 / 回归 / 双路由一致） |
| 冻结合规 | ✅ 不触发红线（详见 §7） |

**最关键的范围澄清（纠正前序假设）**：前序探索曾假设 M9-001 会修改 `ContinueExploringPanel.tsx`。但 M9-001 设计规范 §4 明确将"修改 frontend 代码"列为 Non-Goal，§11 明确"前端集成边界仅供 M9-002 参考"。因此本计划将 **M9-001 限定为后端引擎 + API**，前端渲染增强为 **M9-002** 范围。这样既尊重已审核的设计边界，也避免 M9-001 触碰冻结红线中的 "Recommendation" 语义争议放大。

```
App.tsx: fetchNode → GET /entity/{id}
        → main.py entity()

            ├─ get_exploration_view()
```

## 5. Development Sequence

> 每步均以其前一阶段产出为输入；全部为 additive，可独立回滚。

**Step 1 — Backend Engine core**
- 输入：`exploration_engine.py` 现状（L97–103 常量、L148–168 dataclass、L411–442 `explore()`）。
- 产出：`recommend_next()` + 常量 + `RecommendationItem`。候选生成 = `explore(gid, max_depth=1)`（覆盖直接 + 跨主题单跳）+ 可选 `explore(gid, max_depth=2)` 中 depth=2 节点（降权）；每候选计算 `RelationshipWeight`(复用 `RELATIONSHIP_MEANING`，非 null `weight` 覆盖) / `TimelineRelevance`(复用 `_temporal_coherence` 公式，半衰期 500) / `ThemeConnection`(同 topic +0.5 / 跨主题边 +0.3 / 共享 labels 封顶 +0.2) / `ExplorationDiversity`(seen 惩罚 0.2 / 类型新颖性 1.0→0.85→0.6)。复合 `RecommendationScore = Σ REC_Wᵢ·分量ᵢ`，排序截断 top N，模板化生成 `reasons[]` 与 `relation_path`。
- 验收：`recommend_next()` 为纯函数；同输入 → 同输出（可快照）。

**Step 2 — Backend Service wrapper**
- 输入：Step 1 的 `recommend_next()`。
- 产出：`KnowledgeService.recommend_next(...)` 返回 `RecommendationResult` dict；`current_entity` 取 `_gg.get_node(gid)` 的 `global_id/name/type`。
- 验收：返回结构符合 M9-001 §10.1 契约；不新增端点。

**Step 3 — Backend API handler + route**
- 输入：Step 2 的 wrapper。
- 产出：`recommend()` handler + `/entity/{entity_id}/recommendations` 在 v1 / legacy 双注册（与 `entity()` 同构，404 校验复用）。
- 验收：`GET /api/v1/entity/{id}/recommendations` 与 legacy `/entity/{id}/recommendations` 响应逐字节一致。

**Step 4 — Tests**
- 输入：Step 1–3。
- 产出：`test_recommend.py`（§6 四类 + 双路由一致）。
- 验收：全部 green；现有 `test_exploration_engine.py` / `test_api_v1.py` 仍 green（回归）。

**Step 5 — Regression & Freeze gate**
- 输入：Step 1–4。
- 产出：运行 `pytest`（backend）、`vitest`（frontend，不受影响）、`scripts/freeze-check.mjs`。
- 验收：backend pytest 全绿、freeze-check D=0、frontend 无变更故 vitest 维持基线。

## 6. Test Strategy

| 测试 | 方法 | 断言 |
|---|---|---|
| **Deterministic** | 固定 `(gid, seen_global_ids, max_results)` 调 `recommend_next()`；快照比对 | 同输入逐字节一致；无随机、无时钟参与排序 |
| **Ranking** | 构造候选 A（直接 `influenced` 边 + 时间相近 + 同 topic）vs B（depth=2、`related_to` 边 + 时间远） | `A.score > B.score` 且 A 排序在前 |
| **Explainability** | 取 top N 每项的 `reasons` / `relation_path` | `reasons` 非空且 ≥1 条；存在直接边时 `relation_path` 非空 |
| **Regression** | 重跑 `explore()` / `find_connections()` 基线快照；`freeze-check.mjs`；全套 pytest + vitest | 引擎旧行为不变；枚举 8/18 不变；全套 green |
| **Route parity** | `test_v1_recommend_matches_legacy`（仿 `test_api_v1.py`） | `GET /api/v1/entity/{id}/recommendations` == legacy 路径响应 |

**测试约定（沿用现有）**：`backend/tests/*.py` 手动 `sys.path` 插入；经 `KnowledgeService(JsonTopicRepository(DATA_DIR))` 构建（`DATA_DIR = backend/data/examples`）；端点测试用 `TestClient(app)`。

## 9. Acceptance Criteria

M9-001 实现（待 PO 批准后）视为完成，当且仅当：

1. `GET /entity/{entity_id}/recommendations?limit=5&seen=` 在 `/api/v1` 与 legacy 路径均可用，响应逐字节一致。
2. 响应符合 M9-001 §10.1 `RecommendationResult` 契约（`current_entity / recommendations / algorithm_version="m9-001.v1" / parameters / metadata`）。
3. 每个 `recommendations[]` 项含 `score`、`score_breakdown`（4 分量）、`reasons[]`（≥1 条）、`relation_path`。
4. 确定性：同 `(gid, seen, params)` → 同输出（快照测试 green）。
5. `explore()` / `find_connections()` 行为与分数零变化（回归测试 green）。
6. `scripts/freeze-check.mjs` D=0；`ENTITY_TYPES`=8 / `RELATIONSHIP_TYPES`=18 不变。
7. backend pytest 全绿；前端零改动（vitest 维持 M9-000 基线）。
8. 未修改 `CURRENT_ARCHITECTURE_BASELINE.md`、未引入新依赖、未 commit/push/tag（除非 PO 另行指示）。
