## 1. Scope

**Goal**: Surface an explainable, retractable **Exploration Journey** in the frontend — annotating each stop with *why it was reached* (the recommendation `reasons` / `relation_path` captured when the user followed a suggestion), so M9-001/002's "explainable next-node" value persists across the whole exploration session instead of vanishing at the first click.

**In Scope (frontend-only)**:
- New component `frontend/src/components/ExplorationJourney.tsx` (3-layer: `buildJourney` pure fn + `ExplorationJourneyView` pure presentational + `ExplorationJourney` container). Pure consumer of App navigation history; owns **no** navigation state.
- App.tsx: new session-scoped `journeyReasons` state (annotation map) + `goHome` reset + additive mount of `<ExplorationJourney>` sibling (after `ExplorationTrail`, before `RecommendationPanel`) + `onNodeClick` 2-arg capture.
- `RecommendationPanel.tsx`: additive `RecommendationContext` type + `buildRecommendationContext` pure fn + backward-compatible `onNodeClick(gid, ctx?)` (1-arg callers unchanged).
- New `.he-journey*` styles appended to `App.css` (reuse `--he-*` tokens; no override of `.he-trail`/`.he-recommend`).
- New tests `frontend/src/components/__tests__/ExplorationJourney.test.tsx` (13 cases).

**Out of Scope (untouched, verified)**: backend / `main.py` / `exploration_engine.py` / `navigation.ts` (semantics) / schema / enums (`ENTITY_TYPES=8`, `RELATIONSHIP_TYPES=18`) / `package.json` / `requirements*` / `.github/` / `scripts/` / `CURRENT_ARCHITECTURE_BASELINE.md` / any new dependency / any new navigation code.

## 3. Integration

- **Navigation (read-only consumer)**: `ExplorationJourney` receives `history` / `cursor` from App and renders a derived `<ol>` path. It **never** calls `pushHistory` / `goTo`; node click → `onStepClick(index)` → App `goTo(history[index])` → existing single navigation path (closed loop, zero new nav code).
- **Annotation capture (recommendation → journey)**: When a `RecommendationPanel` card is clicked, `onNodeClick(gid, buildRecommendationContext(rec))` carries the recommendation's `reasons` / `relation_path` / `score` / `candidateSource`. App, **only if `ctx` present**, writes `journeyReasons.set(gid, {fromName, relationPath, reasons, score, candidateSource, capturedAt})`. Direct navigation (no ctx) leaves the annotation `null`. This map **never enters `navigation.ts`** — it is render-layer enrichment only.
- **Reset boundary**: `goHome` resets `journeyReasons = new Map()`; annotations are session-scoped (lost on refresh), keeping the navigation truth clean.
- **Existing `ExplorationTrail`**: **unchanged** — Journey is a parallel sibling; both derive from the same `history`/`cursor`.

## 4. UX Flow

Order in the entity view (additive sibling):
1. **What it is** — `EntityPage`
2. **Where you've been (footprint)** — `ExplorationTrail` (unchanged)
3. **Your Exploration Journey (with why)** — `ExplorationJourney` (NEW): titled "Your Exploration Journey"; each stop shows type + name; stops reached via a recommendation show a `via <fromName>` why-block with `reasons[]` + `relation_path[]` (relationship + direction). Current stop marked `is-current` (`aria-current="step"`).
4. **Where to go next, and why** — `RecommendationPanel` (unchanged behavior; now also feeds Journey annotations)
5. **Where else you can go** — `ContinueExploringPanel` (unchanged)

States:
- **No journey** (`history.length < 2`) → renders `null` (consistent with `ExplorationTrail`).
- **Stop without annotation** (direct navigation) → renders name only, no why-block.
- **Click** → `onStepClick(index)` → App `goTo` (single path, no duplicate nav logic).

## 5. Tests

7. View renders full path with current marker, no why block when unannotated.
8. View injects the recommendation payload (reasons + relation path + `via fromName`).
9. View degrades gracefully when a node has no payload (label still shows).
10. Container renders exactly one button per history node — never a second navigation tree.
11. Container wires `onStepClick` to each node index (structural aria contract "Return to"/"Current:" matching `goTo`).
12. `buildRecommendationContext` extracts the why fields from a recommendation item.
13. `RecommendationPanelView` 1-arg `onNodeClick` still works (backward-compatible markup unchanged).
