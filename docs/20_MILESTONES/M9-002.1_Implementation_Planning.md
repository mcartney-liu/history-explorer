## 1. Executive Summary

M9-002 delivers the **RecommendationPanel**: a frontend-only React component that consumes the already-shipped deterministic Next-Node Recommendation Engine (`GET /entity/{id}/recommendations`, landed in v0.11.0 / M9-001.2) and renders its results as a "where to go next, and why" discovery zone in the entity view.

The work is **pure presentation + one self-fetching container**, mounted as a new sibling in the entity branch of `App.tsx` (between `EntityPage` and `ContinueExploringPanel`). It introduces **no algorithm changes, no API contract changes, no backend changes, no schema changes, no new dependencies, and no new global state**.

This plan was derived from a read-only deep-dive into the current frontend architecture (Phase 0–5). All findings below are grounded in actual code reads, not assumptions.

**Scope at a glance**
- **In Scope**: 1 new component file, 1 additive mount in `App.tsx`, 1 CSS namespace addition in `App.css`, 1 new test file.
- **Out of Scope**: backend, schema, `package.json`/deps, `CURRENT_ARCHITECTURE_BASELINE.md`, any new navigation path, any new API client module, caching/state libraries.

Alignment conclusions:
1. **Implementation must flow API → React UI → exploration loop.** No algorithm or API-contract change is permitted (engine already shipped in v0.11.0).
2. **API contract to consume** (`GET /entity/{id}/recommendations?limit=5&seen=comma,sep`):

   - Empty `seen` is safe.
3. **Design decision ratified**: component self-fetches + holds local 3-state (loading / error / loaded). Presentational cards show `reasons` + `relation_path`. Click → existing `openEntity(global_id)` (no new nav code).
4. **Mount strategy = Candidate C (independent sibling)**: `RecommendationPanel` is a sibling in the entity Fragment, NOT injected into `EntityPage` (rejected A) and NOT merged into `ContinueExploringPanel` (rejected B — that panel's contract is explicitly "NOT a recommender").

## 6. Component Design

### 6.1 Layering (mandatory for testability)
```
RecommendationPanel (container)
  ├─ useState: { status: 'loading'|'error'|'loaded', data?, errorKind? }
  ├─ useEffect: on mount / entityId change → fetchRecommendations(...)
  └─ renders <RecommendationPanelView .../>   (pure, props-only)

fetchRecommendations(entityId, { limit, seen }, signal?)  → pure async helper
  - GET `${API_BASE}/entity/${encodeURIComponent(entityId)}/recommendations?limit=${limit}&seen=${seen}`
  - returns parsed RecommendationResult or throws typed ErrorKind ('network'|'notfound'|'parse')

RecommendationPanelView (pure presentational)
  - props: status, data?, errorKind?, seenGlobalIds?, onNodeClick?
  - loading   → <LoadingSkeleton label="推荐探索路径" />
  - error     → <ErrorCard kind={errorKind} />
  - loaded & empty → null
  - loaded    → cards
```

**Why this split**: `vitest` runs `environment: 'node'` and existing tests use `renderToStaticMarkup`, which does **not** run `useEffect`. Separating the fetch helper (testable via mocked `global.fetch` in node) from the view (testable via `renderToStaticMarkup` with preloaded props) is the only way to get real coverage without introducing jsdom.

### 6.2 API_BASE reuse
Recommendation: inline the identical expression inside `RecommendationPanel.tsx`:
```ts
const API_BASE: string = import.meta.env.VITE_API_BASE || 'http://localhost:8000'
```
This is **zero new-file, zero App.tsx export churn**, and is byte-identical to `App.tsx` L54 (no behavior change). *Alternative (PO option)*: export `API_BASE` from `App.tsx` and import it — slightly more DRY but widens `App.tsx`'s exported surface. Recommended default = inline.

### 6.3 Props contract (mirrors `ContinueExploringPanel`)
```ts
type RecommendationPanelProps = {
  entityId: string
  seenGlobalIds?: Set<string>      // only marks 'is-seen', never reorders
  max?: number                     // default 5
  onNodeClick?: (globalId: string) => void   // wired to openEntity
}
```
- `localName(gid)` pure helper reused (identical to `ContinueExploringPanel`) for display fallback.
- Card click → `onNodeClick?.(target.global_id)` → `openEntity(global_id, localName(global_id))` → existing `navigateTo` → `fetchNode` → `GET /entity/{id}`. **Exploration loop closes with zero new nav code.**

### 6.4 Card content
Each recommendation card shows:
- `target_entity.name` (+ `localName` fallback) and `target_entity.type` badge.
- `score` (rounded to 4 dp, mirrored from engine).
- `reasons[]` — bullet list of human-readable "why".
- `relation_path[]` — compact chips: `from →(relationship)→ to` with direction + weight.
- `is-seen` marker when `target.global_id ∈ seenGlobalIds` (visual only).

## 7. Data Flow

1. `App` entity branch renders `<RecommendationPanel entityId={current.id} seenGlobalIds={seenGlobalIds} max={5} onNodeClick={openEntity} />`.
2. On mount / `entityId` change, container `useEffect` calls `fetchRecommendations(current.id, { limit: 5, seen: [...seenGlobalIds].join(',') })`.
3. Helper issues `GET /entity/{id}/recommendations?limit=5&seen=...`; parses `RecommendationResult`.
4. Container `setState` → `loaded` with `recommendations[]`; view renders cards.
5. User clicks a card → `onNodeClick(global_id)` → `openEntity` → `navigateTo` → `fetchNode` → `GET /entity/{id}` (re-render of entity view, incl. a fresh RecommendationPanel for the new entity). **Closed loop, no new route.**

## 8. Test Strategy (Phase 5)

Hard constraint: **no jsdom**. All tests run under `environment: 'node'` with `renderToStaticMarkup` and/or mocked `global.fetch`.

### 8.1 Pure fetch helper (`fetchRecommendations`)
Node-unit, mock `global.fetch`:
- success → returns parsed `recommendations[]` (assert shape: `target_entity.global_id`, `score`, `reasons`, `relation_path`).
- 404 → throws `ErrorKind 'notfound'`.
- network failure (fetch rejects) → throws `ErrorKind 'network'`.
- malformed JSON → throws `ErrorKind 'parse'`.
- empty `seen` string is passed through safely.

### 8.2 Presentational view (`RecommendationPanelView`)
`renderToStaticMarkup` with preloaded props:
- `loading` → renders `LoadingSkeleton` markup.
- `error` → renders `ErrorCard` markup with correct kind.
- `loaded` + empty → renders `''` (null, consistent with `ContinueExploringPanel`).
- `loaded` + items → renders N cards in engine order (no reorder); `reasons` + `relation_path` present; `is-seen` class applied when `global_id ∈ seenGlobalIds`.

### 8.3 Interaction
- Click a card → assert `onNodeClick` called with `target.global_id` (verified via a thin click-handler unit using a mock callback; note: full DOM-click simulation requires jsdom which is unavailable, so interaction is validated at the callback-binding level + a manual dev-server check).
