## 1. Executive Summary

M9-001.2 在冻结架构边界内，于确定性的 `ExplorationEngine` 之上实现了 **Deterministic Next-Node Recommendation Engine**，并作为 additive 层接入 REST API。它从"用户当前所在实体"出发，产出一组**可解释、可测试、可复现**的"下一站"推荐，使产品 DNA 定义的 "Infinite Exploration" 形成闭环。

本实现**不**引入任何 AI / LLM / 图数据库 / 新依赖 / 前端改动：完全复用冻结评分原语（`RELATIONSHIP_MEANING`、`TYPE_IMPORTANCE`、时间半衰期 500），以独立的 `REC_W_*` 权重组合成推荐分数。所有变更落在 4 个允许修改的文件内，现有 `explore()` / `find_connections()` 行为与分数未被修改。

## 3. Architecture Impact

### 3.1 新增结构（均位于 `exploration_engine.py`，additive）

- **常量**：`DEFAULT_MAX_RESULTS = 5`、`REC_W_RELATIONSHIP=0.40`、`REC_W_TIMELINE=0.25`、`REC_W_THEME=0.20`、`REC_W_DIVERSITY=0.15`、`ALGORITHM_VERSION="m9-001.v1"`。
  - 这些 `REC_W_*` 是 M9-001 **新增**独立权重，**未**覆盖冻结的 `W_RELATIONSHIP=0.35` 等。
- **`RecommendationItem`** dataclass：`target_entity` / `score` / `score_breakdown`(4 分量) / `reasons` / `relation_path` / `metadata`。
- **`RecommendationResult`** dataclass：`current_entity` / `recommendations` / `algorithm_version` / `parameters` / `metadata`（M9-001 §10.1 契约）。
- **`recommend_next(gid, seen_global_ids=None, max_results=5)`** 方法：
  1. 候选生成：复用 `explore(gid, max_depth=2)`（直接 + 二跳，确定性，无新增遍历）。
  2. 四分量（独立、纯函数）：
     - `RelationshipWeight` = 路径首跳边的 `RELATIONSHIP_MEANING`（复用冻结查表，[0.4,1.0]）。
     - `TimelineRelevance` = `max(0.5, _temporal_coherence(...))`（复用冻结公式，半衰期 500，钳至设计值域 [0.5,1.0]）。
     - `ThemeConnection` = 同 topic +0.5 / 跨主题 +0.3 / 共享 labels 封顶 +0.2（[0,1.0]）。
     - `ExplorationDiversity` = 已访问 0.2 / 类型新颖 1.0→0.85→0.6。
  3. 确定性贪心多样性选择（按 base 分数降序 + gid 升序初始排序；按已选类型数算 diversity）。
  4. 最终按复合分数降序 + gid 升序排序截断至 `max_results`。
- **理由生成** `_build_reasons()`：模板化（M9-001 §9 规则），**无**任何生成式模型调用。
- 时间戳 `metadata.generated_at` 仅作信息性 metadata，**不参与排序**（保证推荐集合本身逐字节确定）。

### 3.2 接入层

- `KnowledgeService.recommend_next()`：**薄封装**，委托 `self._exploration_engine.recommend_next()` 并返回 `RecommendationResult`，现有 API 契约未变。
- `main.recommendations()` handler + 双路由：同一函数同时挂载于 `/api/v1/entity/{id}/recommendations` 与 legacy `/entity/{id}/recommendations`（operation_id 区分），保证 **v1 == legacy**。支持查询参数 `limit`（默认 5）与 `seen`（逗号分隔的已访问 global_id，用于多样性惩罚）。404 当实体/其 global_id 缺失。

### 3.3 不变项（显式确认）

- `explore()` / `find_connections()` 函数体与冻结 `W_*` 评分公式**未触碰**。
- `ExploredNode` / `PathStep` 等既有契约**未改**。
- 前端**零改动**（M9-001 设计规范 §11 明确禁止改前端；前端消费归 M9-002）。

`scripts/freeze-check.mjs` 的 **scope 守卫**由环境变量 `FROZEN_SCOPE` 控制（默认 `frontend`，见脚本 L34/L36 "adjust per milestone"）。M9-001 是 **PO 已批准的 backend 里程碑**，故 CI / 本地校验须以 `FROZEN_SCOPE=backend` 运行以放行 backend 变更；**实质性守卫（token / dependency / enum）在任何 scope 下均生效**。

- `FROZEN_SCOPE=frontend` 下：scope 守卫会报 2 条 D-class（因 backend/app 被改）——这是**按里程碑 scope 的预期行为**，非代码违规。
- `FROZEN_SCOPE=backend` 下：**PASSED — no D-class violations**（token/dependency/enum 全部通过，枚举 8/18 确认）。

> **建议（供 PO）**：M9-001 的 CI job 应显式设置 `FROZEN_SCOPE=backend`；scope 守卫是"按里程碑放行 backend"的开关，不是冻结语义本身。冻结语义（禁 AI/LLM/图库/新依赖/枚举变更）由其余三项守卫在任意 scope 下持续强制。
