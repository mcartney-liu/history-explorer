## 1. 当前架构真实评估（阶段一）

### 1.1 目前 Knowledge Layer 已具备什么能力？

基于 `backend/app/core/` 真实代码：

| 能力 | 模块 | 真实状态 |
|---|---|---|
| 存储无关数据访问 | `repository.TopicRepository` + `JsonTopicRepository` | 读 `data/examples/*_example.json`，进程内缓存（`main.py:73`） |
| 全局注册表 | `registry.KnowledgeRegistry` | 四索引：Entity（`topic→{id→entity}`）、Global（`global_id→EntityRef`）、Alias（`alias→[refs]`）、Topic（`topic→meta`）；`resolve()` 支持 local/global_id 互解（`registry.py:82-109`） |
| 单主题图 + 遍历 | `graph.DirectedGraph` + `KnowledgeGraph` | per-topic 邻接；`neighbors(out/in/both)`、`bfs`、`find_path`、`orphans`、`cycles`（`graph.py:28-170`） |
| 全局 id 跨主题解析 | `registry` + `validation._validate_cross_topic` | `find_by_global_id('rome:civ-roman')` 可解析；跨主题边不再误判 dangling（`validation.py:580-632`） |
| 搜索 | `search.build_search_index` + `SearchProvider` | exact/alias/contains 三档确定性排名（`search.py:23-122`） |
| 时间索引 | `timeline.TimelineIndex` | 正规化 + 按年查询 `get_by_year`/`get_range`（`timeline.py`） |
| 投影 | `exploration` | `build_exploration_view`/`build_exploration_response`，纯函数（`exploration.py`） |
| Facade | `knowledge_service.KnowledgeService` | 只聚合、零自有算法（公共方法 18 个，见 `knowledge_service.py:47-155`） |

**已落地的事实数据**（`/health` readiness，来自真实数据文件）：
```
Topics: 4 | Entities: 26 | Relationships: 29 | Timeline: 7
Warnings: 0 | Errors: 0 | Status: healthy
```
4 主题 = `roman_empire`(6 ent/7 rel)、`egypt_technology_religion`(6/5)、`hellenistic_world`(8/11)、`silk_road`(6/6)。**6 条跨主题边**全部用 `namespace:id` 表达。

### 1.2 目前缺少什么能力？

1. **跨主题图（GlobalGraph）缺失**——这是 M3-003 明确发现并记入 `SUGGESTIONS.md §O7` 的边界。`KnowledgeGraph` 为**每主题一张图**；跨主题 `global_id` 边在构图时被跳过（"dropped as dangling-by-design"，`graph.py:161-166`）。因此 `find_related`/`get_graph` 返回的是**单主题邻居**。
2. **探索「路径」生成能力缺失**——当前 `/entity` 返回的是 `relationships` 扁平列表 + `exploration.related_entities`（单主题）。没有「从 A 走 N 步通往 B 的探索路径」概念，也没有节点排序/推荐。
3. **跨主题 `other` 补全缺失**——`get_entity_relationships`（`knowledge_service.py:106-133`）遍历本主题 `relationships`，跨主题边的另一端不在本主题 `entity_by_id`，故 `other` 解析为空（`{id:None, name:'', type:''}`）。前端因此**看不到跨主题邻居**。
4. **关系权重/时间排序**——schema 已预留 `weight`/`confidence`/`valid_time`（全部 optional，当前数据均为 `null`），但没有消费它们的排序逻辑。
5. **Graph UI**——M3-001/002/003 均未做可视化（记入 §O4）。

### 1.3 为什么 M3-003 后仍然无法形成真正的跨文明探索？

**数据已互联，但「图遍历」与「前端展示」两个环节断裂：**

- **数据层**：6 条跨主题边真实存在（如 `roman_empire:civ-roman → influenced → hellenistic_world:civ-greek`）。
- **注册表层**：`find_by_global_id` / `resolve_entity` 能解析任意跨主题引用。M3-003 测试 `test_rome_reaches_egypt_and_china_via_network` 已证明从 `roman_empire` 经跨主题边 BFS 可达其余 3 个主题。
- **断裂点 1（Graph）**：`KnowledgeGraph` 在构图时把跨主题边丢弃（`graph.py:161-166`），所以 `neighbors('civ-roman')` 不会返回 `hellenistic_world:civ-greek`。
- **断裂点 2（投影）**：`get_entity_relationships` 不补全跨主题 `other`，前端拿不到可点击的跨主题目标。
- **断裂点 3（前端认知）**：前端 `RelatedEntityList`/`RelationshipView` 渲染的是 `exploration.related_entities`（单主题内），没有「这是个跨主题文明」的概念。

**结论**：跨文明探索目前停在两步——「数据说世界是连的」+「注册表能解析连接」，但用户点击罗马后，**界面不会呈现埃及/希腊/丝路作为可跳转的邻居**。这正是 M3.5 要补的闭环。

### 1.4 当前 graph.py / KnowledgeService 是否适合演进为 GlobalGraph？

**结论：适合，且应当是 additive 演进，不是重写。**

- `DirectedGraph`（`graph.py:28-126`）是**与主题无关的纯邻接结构**（nodes 是字符串 id，edges 带 type）。它完全有能力承载「全局节点图」——只需把节点从「local id」升级为「global_id 形式的稳定 key」，再把跨主题边纳入。
- `KnowledgeGraph` 的 per-topic 构建逻辑（`graph.py:136-167`）已经写好了 `global_id → local_id` 的解析骨架（`_resolve`）。演进 GlobalGraph 只需：① 让跨主题边的端点也通过 `registry.resolve` 解析为全局 key；② 把多张 per-topic 图合并为一张全局图。
- `KnowledgeService` 已是 Facade，**增加** `find_related_global` / `explore_path` 等方法不会破坏现有 18 个公共方法（不动、不删）。
- 风险极低：新增 `core/global_graph.py` 与 `core/exploration_engine.py` 两个**新模块**，`KnowledgeService` 仅新增委托方法。现有 `graph.py`/`get_graph`/`find_related` 全部保留（向后兼容，其他测试不破）。

## 2. GlobalGraph 设计（阶段二 · 一）

### 2.1 当前 schema 是否支持？（冲突点必须指出）

**部分支持。用户设计的 Node/Edge 类型中有 4 项超出当前 v2 schema，必须显式处理。**

| 用户设计 | 当前 schema（v2，`validation.py:26-57`） | 冲突 |
|---|---|---|
| Node: Civilization / Person / Event / Technology | ✅ 在 7 active types 内 | 无 |
| Node: **Place** | ⚠️ schema 用 `Location`（语义等价，命名不同） | 命名差异，建议**沿用 schema 的 `Location`**，不新增 `Place`（避免 type 爆炸） |
| Node: **Idea** | ❌ 不在 active，reserved 也没有 `Idea`（reserved: Artifact/Institution/Culture/Language） | **需 additive 新增 `Idea` 到 `ENTITY_TYPES`** |
| Edge: influenced / caused / traded_with | ✅ 在 15 types 内 | 无 |
| Edge: **contemporaneous** | ⚠️ schema 是 `contemporary_with`（拼写不同） | **沿用 schema 的 `contemporary_with`**，不新造 `contemporaneous` |
| Edge: **inherited** / **conquered** / **spread** | ❌ 不在 15 types 内 | **需 additive 新增这 3 个到 `RELATIONSHIP_TYPES`**（建议补充语义：`inherited`=文明/技术传承，`conquered`=征服，`spread`=传播扩散） |

> **架构纪律声明**：以上新增**一律走 additive enum 扩展**（在 `validation.py` 的 `ENTITY_TYPES`/`RELATIONSHIP_TYPES` 加项），**绝不重命名** `Location`→`Place` 或 `contemporary_with`→`contemporaneous`——重命名会破坏已有数据与校验契约。这是本轮评审对设计冲突的明确修正。

### 2.2 哪些字段需要新增？

**无需新增字段到 Entity/Relationship 的「形状」。**

- Entity：当前 `id/type/name/description/global_id/aliases/labels/start_date/end_date/...` 已足够描述 Node。新增 `Idea` 类型只是 enum 扩展，字段复用 BaseEntity + 未来可选 `texts`（Religion 已有先例）。
- Relationship：当前 `source/target/type/confidence/citation/evidence/valid_time/weight` 已完整。GlobalGraph 仅消费 `source/target/type/weight`；`weight` 已预留但当前为 `null`，**Exploration Engine 用默认权重策略兜底**（见 §3.4）。
- 全局 key：GlobalGraph 的内部节点标识建议用 `global_id`（如 `roman_empire:civ-roman`），因为 `global_id` 已是 `namespace:id` 全局唯一且不出现在 URL 中（schema §5）。**不改 `id` 字段。**

### 2.3 是否应该修改现有 Entity / Relationship？

**不应该修改字段形状。** 现有 26 实体、29 关系的字段全部保留。仅：
- additive 扩展 `ENTITY_TYPES`/`RELATIONSHIP_TYPES`（容纳 `Idea`/`inherited`/`conquered`/`spread`）。
- 数据层面（M3.5 实施时）可**新增**关系使用新类型，但**不修改**既有关系的 `type`（除非确实是数据纠错，且需重建校验）。

### 2.4 如何保持向后兼容？

- **数据兼容**：v2 schema 全 optional + additive；新 enum 项不影响旧数据。
- **API 兼容**：GlobalGraph 是 Knowledge Layer 内部模块，不直接暴露为端点。若要让前端看到跨主题邻居，采用 **additive 投影补全**——在 `get_entity_relationships` / `get_exploration_view` 的 `other` 中填充 `global_id` + `topic` 字段（新增字段，不删现有 `id/name/type`）。或直接新增 **additive 端点** `/api/v1/explore/{id}/path`（见 §2.6）。
- **图兼容**：保留 `KnowledgeGraph`（per-topic）与 `get_graph`/`find_related` 不变；新增 `GlobalGraph` 与 `find_related_global` 并行。

### 2.5 GlobalGraph 应该属于 Knowledge Layer 还是独立 Service？

**结论：属于 Knowledge Layer（作为 `core/` 内新模块），不是独立 Service。**

理由：
1. M3-001 架构已明确「Knowledge Layer = registry / graph / search / timeline / exploration」，Graph + Traversal 本就是 Knowledge Layer 的职责（`__init__.py:6-13`）。GlobalGraph 是「图」的自然扩展。
2. 用户给的目标结构 `Frontend → API → Exploration Service → GlobalGraph → Knowledge Layer → Repository` 中，GlobalGraph 与 Exploration Service 都位于「Knowledge Layer 之上、API 之下」——这与「二者都是 Knowledge Layer 内部模块」不冲突，只是层级表述。建议映射为：
   - `core/global_graph.py` —— GlobalGraph（构建全局节点图 + 全局邻接/最短路径）。
   - `core/exploration_engine.py` —— Exploration Service（确定性路径发现/排序/推荐，纯算法，依赖 GlobalGraph）。
   - `KnowledgeService` 暴露 `find_related_global` / `explore_path` 作为稳定 API。
3. 独立成「Service」会破坏 M3-001 的「KnowledgeService 是 API 唯一对话对象」约定，且与 Repository 解耦原则无收益（GlobalGraph 仍消费 registry，与存储无关）。

**未来切 Neo4j 的接缝已就位**：Repository 抽象（`TopicRepository`）只决定「数据从哪来」；GlobalGraph 构建自 registry/graph，与存储无关。切 Neo4j 时只需新增 `Neo4jTopicRepository` 替换组合根一处（`main.py:73`），GlobalGraph/Exploration 逻辑零改动。**无需为 Neo4j 新预留接口。**

### 2.6 技术设计：GlobalGraph 如何演进？


## 3. Exploration Engine 设计（阶段二 · 二）

目标：用户点击 `Roman Empire`，不返回「related_entities 列表」，而生成**探索路径**。

```
Rome → Egypt → Alexander → Greece → Persia   (用户示例)
```

### 3.1 设计原则（V1）

- **不使用 AI**：纯规则 + 确定性评分（与 M2 冻结原则一致，schema §8）。
- **确定性**：无随机、无网络调用、相同输入必然相同输出（可单测）。
- **Additive**：消费 GlobalGraph，产出「路径/推荐」，不修改既有图。
- **可解释**：每条推荐带 `reason`（关系类型 + 时间/权重），前端可展示「为何推荐这条路」。

### 3.2 第一版算法（伪代码）

```
explore_path(start_ref, goal_ref=None, max_depth=3, top_n=5):
    start_gid = registry.resolve(start_ref).global_id
    graph = global_graph
    # 1) 路径发现：若给定 goal，用 find_path 求最短确定性路径
    if goal_ref:
        path = graph.find_path(start_gid, goal_gid)   # 确定性最短路径
        return render_path(path)
    # 2) 否则生成「下一步推荐」：从 start 出发，评分排序取 top_n
    candidates = []
    for edge in graph.neighbors(start_gid, "both"):
        other = edge.other_global_id
        score = score_node(start_gid, other, edge)
        candidates.append((score, other, edge))
    candidates.sort(by=score, stable=True)   # tie-break: global_id 字典序（确定）
    return {
        "start": start_gid,
        "recommendations": [render(other, edge, score) for _,other,edge in candidates[:top_n]],
        "paths": derive_paths(start_gid, top_n)   # 每条推荐的 1~2 步预览
    }
```

### 3.3 节点排序（score_node）

综合四维，输出稳定分数：

```
score = W_TYPE * type_weight(edge.type)
      + W_TIME * time_proximity(start, other)
      + W_ENTITY * entity_type_pref(other.type)
      + W_DIR * direction_weight(edge)        # outgoing 略高于 incoming
```

- **type_weight**（关系类型优先级）：`caused`/`influenced`/`ruled`/`invented` = 1.0；`part_of`/`traded_with`/`practiced` = 0.7；`related_to`/`contemporary_with` = 0.5；新类型 `inherited`/`conquered`/`spread` 落地时给 0.7~0.9（在 Engine 内常量表，集中可调）。
- **time_proximity**：用 `start_date.value`（BCE 为负、CE 为正）。`abs(start.value - other.value)` 越小分越高；无时间则中性 0。容错：`approximate` 精度降级但不报错。
- **entity_type_pref**：`Civilization`/`Person`/`Event` = 1.0（探索主线）；`Technology`/`Religion` = 0.8；`Location` = 0.6；`Time Period` = 0.3（避免时间锚点刷屏）。
- **direction_weight**：outgoing = 1.0，incoming = 0.8。

权重 `W_*` 默认 `{TYPE:0.4, TIME:0.3, ENTITY:0.2, DIR:0.1}`，集中在 Engine 常量，便于调参（不进 schema、不改数据）。

### 3.4 关系权重（relationship.weight）

- 当前数据 `weight` 全 `null`（schema 预留但未填）。
- V1 策略：**若 `weight` 非空则用，否则用 type_weight 默认表**（§3.3）。
- 不要求 M3.5 回填权重数据；未来数据可渐进补 `weight`/`confidence`，Engine 自动采用。

### 3.5 时间因素

- 用 `start_date.value`（已正规化）。
- 时间排序方向：默认「时间接近优先」；若关系为 `caused`（因果），路径按 `caused` 箭头方向体现先后（被 type_weight 自然加权）。
- 跨 BCE/CE 边界：`abs` 差值计算已是连续数轴（-27 与 14 差 41 年），无特殊边界 bug。

### 3.6 推荐规则示例（确定性）

从 `Roman Empire`（start）生成 top-3 推荐（基于当前数据真实边）：

| 推荐目标 | edge | type_weight | 说明 |
|---|---|---|---|
| Hellenistic Greek World | `influenced`(out) | 1.0 | 罗马受希腊影响（跨主题 global_id 边） |
| Silk Road | `traded_with`(out) | 0.7 | 罗马经丝路贸易（跨主题） |
| Roman Egypt (Aegyptus) | `part_of`(out) | 0.7 | 单主题内，罗马含埃及省 |

→ 前端呈现「下一步可探索：希腊化世界 / 丝绸之路 / 罗马埃及」，每条带 reason，点击即跳（global_id → `/entity/{global_id}`，后端 resolve 已支持）。

## 4. 技术风险

| # | 风险 | 等级 | 缓解 |
|---|---|---|---|
| R1 | **数据规模太小**：26 节点/6 跨主题边，Exploration Engine 路径价值有限，易显「空」 | 中 | M3.5-001 实施时**并行扩充数据**（新增 1~2 主题或更多跨主题边），但不强求；Engine 设计不依赖规模 |
| R2 | **跨主题 `other` 补全影响前端契约**：若 `/entity` 响应 `other` 新增 `global_id`/`topic` 字段，前端需能消费（目前 `other.id` 仅传本地 id） | 低 | 前端 `openEntity(id)` 已把任意 id 传给 `/entity/{id}`，后端 `resolve_entity` 支持 global_id；**前端几乎零改**，仅需确认 breadcrumb 显示 global_id 时回退到 name（已有 `entityNameById`） |
| R3 | **时间数据精度参差**：大量 `approximate`、部分实体无 `start_date` | 低 | Engine 对缺失时间给中性分，不报错 |
| R4 | **Additive 边界漂移**：扩展 `/entity` 响应或新增端点时误伤 legacy | 中 | 严守「只加字段、不加必填」；新增端点走 `/api/v1` 前缀（legacy 不动）；用 `test_explore.py`/`test_search_entity.py` 契约测试守护 |
| R5 | **enum 扩展污染**：新增 `Idea`/`inherited`/`conquered`/`spread` 若命名不当，未来与 Neo4j 映射冲突 | 低 | 与现有 15 类型一致风格（snake_case、动词/名词清晰）；先定 enum freeze 再写数据 |
| R6 | **God Service 回归**：`KnowledgeService` 若把排序/推荐算法写进 Facade | 高(纪律) | Engine 算法全部在 `core/exploration_engine.py`，KS 只委托（延续 M3-001 防 God Service 原则） |

## 5. 新 Roadmap（阶段三）

### 5.1 原计划 vs 调整建议

原计划 `M3-004 Search v2 → M3-005 UI Depth → M3-006 Release`。

**调整建议**：M3-005 的「Graph UI」与 M3.5 的「Exploration Engine」强耦合——Graph UI 需要 Exploration Engine 提供路径数据。**把 M3-005 拆并为 M3.5-001~004**，Search v2（M3-004）保持独立可并行。

### 5.2 新 Checkpoint 拆解

#### M3.5-001 GlobalGraph Foundation
- **目标**：在 `core/` 新增 `global_graph.py`，构建跨主题全局节点图；`KnowledgeService` 增加 `find_related_global`。闭环 §O7。
- **修改范围**：新增 `core/global_graph.py`；`knowledge_service.py` 加 1~2 个委托方法；`test_core.py` 加全局图测试。**不动** `graph.py`/`repository`/`main.py` 公共行为。
- **文件范围**：`backend/app/core/global_graph.py`、`backend/app/core/knowledge_service.py`、`backend/tests/test_core.py`。
- **依赖**：无（独立起步）。
- **风险**：低。
- **验收**：`find_related_global('roman_empire:civ-roman')` 返回希腊/丝路等跨主题邻居；测试证明 Rome→Egypt/China 经图可达。
- **建议并行**：Data Agent 在此阶段**冻结 enum 扩展清单**（`Idea`/`inherited`/`conquered`/`spread`），并可选补 1~2 主题数据（R1）。

#### M3.5-002 Exploration Engine
- **目标**：新增 `core/exploration_engine.py`，实现 §3 确定性路径发现/排序/推荐/时间/权重。
- **修改范围**：新增模块；`KnowledgeService.explore_path` 委托；新增 `test_exploration_engine.py`（纯函数单测，确定性断言）。
- **文件范围**：`backend/app/core/exploration_engine.py`、`backend/app/core/knowledge_service.py`、`backend/tests/test_exploration_engine.py`。
- **依赖**：M3.5-001（必须串行）。
- **风险**：中（排序规则需调参，但确定性可测）。
- **验收**：同输入同输出；`explore_path('roman_empire:civ-roman')` 返回 top-N 推荐带 reason；`explore_path(start, goal)` 返回最短路径。

#### M3.5-003 Exploration API
- **目标**：additive 暴露探索能力。两种可选方案（**推荐方案 A**）：
- **方案 A（推荐，最小侵入）**：在 `get_entity_relationships`/`get_exploration_view` 的 `other` 中 **additive 补全** `global_id`+`topic` 字段；前端无需改 fetch 即可获得跨主题可点击邻居。 legacy + v1 均受益。
- **方案 B**：新增端点 `GET /api/v1/explore/{id}/path`（返回 `{start, recommendations, paths}`）。更纯粹，但前端需新 fetch。
- **修改范围**：`exploration.py` 投影补全（A）或 `main.py` 加路由（B，additive）；`SearchProvider` 不动。
- **文件范围**：`backend/app/core/exploration.py`、`backend/app/main.py`（B 时）、`backend/tests/test_api_v1.py`。
- **依赖**：M3.5-002（串行）。
- **风险**：中（A 需确认前端契约；B 需前端新消费）。
- **验收**：`/entity/{global_id}` 响应含跨主题 `other.global_id`；或 `/api/v1/explore/{id}/path` 返回结构稳定；既有契约测试不破。

#### M3.5-004 Consumer UI Prototype
- **目标**：前端展示跨主题邻居 + 探索路径（轻量，复用 `RelationshipView`/`RelatedEntityList`，additive）。
- **修改范围**：`App.tsx`/`EntityPage.tsx` 消费 `other.global_id`（方案 A 时几乎零改）；或新增 `ExplorationPath.tsx`（方案 B）。**不重写现有组件**。
- **文件范围**：`frontend/src/components/*`（additive）。
- **依赖**：M3.5-003 API contract 冻结（可先用 mock 并行，真实联调串行）。
- **风险**：中（前端体验需打磨）。
- **验收**：点击罗马可见埃及/希腊/丝路为可跳转邻居；vitest + build 绿。

#### 并行保留项：M3-004 Search v2
- 独立于 Exploration（SearchProvider 已可替换）。建议在 M3.5-004 之后做，使「搜索 + 探索」体验连贯；技术上可与 M3.5-001/002 并行（不同模块、无冲突）。

#### 收尾：M3-006 Release & CI

## 6. 多 Agent 可行性分析（阶段四）

### 6.1 角色拆分

| Agent | 负责 | 不碰 |
|---|---|---|
| **Backend Agent** | `core/global_graph.py`、`core/exploration_engine.py`、`KnowledgeService` 委托方法、`exploration.py` 投影补全、`main.py` additive 路由、后端测试 | 前端、数据文件内容（仅消费 enum 清单） |
| **Data Agent** | 冻结/扩展 schema enum（`Idea`/`inherited`/`conquered`/`spread`）、新增/扩主题数据、跨主题边 | 后端代码、前端代码 |
| **Frontend Agent** | `M3.5-004` UI（消费 global_id、路径组件）、前端测试 | 后端、数据 |

### 6.2 可并行 vs 必须串行

**必须串行（强依赖链）：**
- Backend 内部：`M3.5-001 → 002 → 003`（GlobalGraph 是 Engine 的输入，Engine 是 API 的输入）。
- Frontend `M3.5-004` 真实联调依赖 `M3.5-003` API contract 冻结。

**可并行：**
- **Data Agent 与 Backend Agent（部分）**：Data Agent 先与 Backend 对齐 **enum 扩展清单**（契约），之后 Data 补数据可与 Backend `M3.5-001` 的实现并行（数据用已有/新 type，新 type 已由 Backend freeze）。
- **M3-004 Search v2** 与 `M3.5-001/002` 完全并行（不同模块、无共享状态）。
- **Frontend Agent 用 mock 并行**：一旦 `M3.5-003` 的响应结构（A 或 B）在规划阶段冻结，Frontend 可基于 mock JSON 并行开发 `M3.5-004`，待 backend 部署后联调。

### 6.3 协同接口（Agent 间契约）

- **Backend ↔ Data**：`RELATIONSHIP_TYPES`/`ENTITY_TYPES` 扩展清单（Backend 冻结 enum，Data 遵守）。
- **Backend ↔ Frontend**：`/entity` 响应的 `other.global_id` 字段（方案 A）或 `/api/v1/explore/{id}/path` 的 JSON 结构（方案 B）。**必须先 freeze 再各自实现**。
- **Frontend ↔ Data**：无直接依赖（前端只消费 API）。

### 6.4 多 Agent 风险

- enum 未 freeze 前 Data 先用新 type → 校验报 `RELATIONSHIP_UNKNOWN_TYPE` warning（不致命，但脏）。**必须 Backend 先定 enum**。
- 前端 mock 与真实响应结构偏差 → 联调返工。**必须在 M3.5-003 规划阶段冻结 contract + 共享样例 JSON**。

## 7. 结论与建议（给翔哥的决策依据）

**当前状态定性**：M3-001→003 已经把地基打到了「数据互联 + 注册表可解析 + 分层清晰」的程度。跨文明探索只差**最后两层**——GlobalGraph（让图跨主题）+ Exploration Engine（让点击生成路径）。这两层**完全在 M3-001 架构预留的接口内**，additive 落地，风险低。

**我的主动建议（带推荐项，供你拍板）：**

1. **是否按 M3.5-001→004 推进？**
   - ✅ **(推荐)** 是，按顺序做。它直接闭环 O7，且不动公共 API、不动 Repository、不引新依赖。
   - ⚪ 否，先补数据规模（R1）再谈 Engine——但 Engine 不依赖规模，可并行补数据，不必阻塞。

2. **M3.5-003 用方案 A 还是 B？**
   - ✅ **(推荐)** 方案 A（additive 补全 `other.global_id`+`topic`）：前端几乎零改、legacy/v1 双受益、最快闭环。
   - ⚪ 方案 B（新增 `/path` 端点）：更纯粹，但前端需新 fetch，工作量略大。

3. **schema enum 扩展（Idea/inherited/conquered/spread）本轮是否引入？**
   - ✅ **(推荐)** 引入，但仅作 enum 扩展（additive），不要求 M3.5 一定用满；为未来数据留口。
   - ⚪ 暂不动 enum，M3.5 只用现有 15 type——更保守，但"Idea"等概念无法表达。

4. **下一步动作**：建议从 **M3.5-001 GlobalGraph Foundation** 开始实施（Backend Agent 单 Agent 即可，Data Agent 并行 freeze enum + 扩数据）。
