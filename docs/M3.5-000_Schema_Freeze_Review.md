## 0. 为什么需要这一道冻结

M3.5-001（GlobalGraph Foundation）及之后的每一个 checkpoint，都依赖三件"提前锁定"的东西：

1. **词汇表**：`ENTITY_TYPES` / `RELATIONSHIP_TYPES` 的最终集合。Backend 要按它实现图结构，Data Agent 要按它写数据，Frontend 要按它渲染标签。
2. **GlobalGraph 数据规则**：跨主题边怎么纳入、节点/边身份是什么、图索引与路径引擎的边界。
3. **Exploration Engine 的产品边界**：它到底是"什么"，不做什么——否则容易滑向 AI 推荐或 GIS，违反 M2/M3 红线。

本轮（M3.5-000）只产出以上三件的**最终冻结值**，不做实现。所有决策都必须能回答一个问题：**"这对使用 History Explorer 的消费者有什么价值？"**

## 1. 消费者产品定位（先定人，再定 schema）

> 这一节是整份冻结的源头。后面的所有词汇与规则都从这一节推导，而不是反过来。

### 1.1 谁是消费者

History Explorer 的消费者不是图数据库工程师，而是：

- **历史爱好者 / 终身学习者**：出于好奇想"搞懂历史是怎么连起来的"。
- **学生 / 教育者**：需要一个能展示"因果与传播"的叙事工具，而非孤立词条。
- **内容创作者**：需要把跨文明、跨时间的线索组织成可讲述的故事。

他们的共同行为模式：**点开一个文明 / 事件 / 人物 → 想知道"它和什么有关、怎么影响到别的文明、后来怎样"**。

### 1.2 Jobs-to-be-done（消费者要完成的任务）

| # | 任务 | 当前能力缺口 |
|---|---|---|
| JT-1 | 从某个实体出发，看懂它与**其他文明**的联系 | M3-003 后有跨主题边数据，但图与前端都不呈现（详见 M3.5 Review §1.2） |
| JT-2 | 追踪一个**概念 / 技术 / 信仰**如何被继承、征服、传播 | 现有关系类型缺少"传承 / 征服 / 传播"语义 |
| JT-3 | 得到**引导式路径**而非扁平列表，降低探索门槛 | 当前只有单跳 related，无多跳发现 |
| JT-4 | 每个推荐都能解释**"为什么"**（可解释性 → 建立信任） | 推荐若黑盒则违反"确定性、可解释"红线 |

### 1.3 产品价值主张（Positioning Statement）

> **History Explorer 把互联的历史知识图谱变成可解释的发现路径**——让好奇的学习者追随思想、技术与事件如何跨越文明与时间，而**不靠算法猜测他们想要什么**。

关键词：**可解释（explainable）**、**确定性（deterministic）**、**跨文明（cross-civilization）**、**时间感知（time-aware）**。这四点直接来自 JT-1~JT-4，也正好对应 M2/M3 已冻结的红线（无 AI / 无 GIS / 确定性）。

### 1.4 Exploration Engine 在产品循环中的位置

M2 schema §7 的循环 `Explore → Connect → Understand → Discover` 映射到 M3.5：

| 阶段 | 含义 | M3.5 落点 |
|---|---|---|
| Explore | 选定入口实体 | 已有 `/entity`、`/explore` |
| **Connect** | 呈现关联与路径 | **GlobalGraph + Exploration Engine（M3.5 核心）** |
| **Understand** | 看懂"为什么连" | 每条边带 `type` + 可解释路径 |
| **Discover** | 下一跳建议 | 确定性排序的推荐（非 AI） |

结论：**Exploration Engine 是 Connect→Understand→Discover 的引擎，不是搜索引擎（M3-004 管搜索），不是地图（M2 红线禁 GIS），不是推荐系统（M2 红线禁 AI 推荐）。**

## 2. 冻结：ENTITY_TYPES

### 2.1 当前真实基线（来自 `backend/app/validation.py:26-36`）

```
Event, Person, Civilization, Location, Time Period, Technology, Religion   # 7 个
```

### 2.2 冻结决策

**保留全部 7 个，additive 新增 1 个：`Idea`。冻结后共 8 个。**

| 类型 | 状态 | 消费者价值（为什么） |
|---|---|---|
| Event | 保留 | 默认探索锚点（schema §2） |
| Person | 保留 | 历史行动者 |
| Civilization | 保留 | 跨文明探索的主干节点 |
| Location | 保留 | 地理归属（M3.5 仍用 `Location`，**不用用户设计稿里的 `Place`**——保契约） |
| Time Period | 保留 | 时间结构 |
| Technology | 保留 | JT-2 传播线索之一 |
| Religion | 保留 | JT-2 传播线索之一 |
| **Idea** | **新增** | **概念追踪**："民主""斯多葛""一神教"等思想如何跨文明传播，是 JT-2 的核心载体。概念本身就是历史学习者最想"追"的东西。 |

**为什么不激活 `Artifact` / `Institution` / `Culture` / `Language`（v2 reserved）**：
- 它们不是 M3.5 v1 消费者旅程的必需入口；
- 激活会迫使 Data Agent 立即补数据，拖慢 M3.5-001 起步；
- 它们未来仍可 additive 加入，**不激活 ≠ 不支持**，schema 已预留扩展位。

**`Idea` 的特殊处理**：引擎对 `Idea` **不做任何特殊逻辑**——它只是另一种节点，和其它 7 类同权参与路径发现。这样新增零成本、零风险，纯粹是给数据作者一个"概念"锚点。

### 2.3 向后兼容

- **Additive**：只增枚举值，绝不重命名、不删除、不改字段形状。
- 现有 7 类数据的校验、API、前端标签**完全不受影响**。
- `Idea` 在 v1 可以一条数据都没有，引擎照常工作。

### 2.4 冻结清单（最终）

```
Event, Person, Civilization, Location, Time Period, Technology, Religion, Idea   # 8 个
```

## 3. 冻结：RELATIONSHIP_TYPES

### 3.1 当前真实基线（来自 `backend/app/validation.py:39-57`）

```
caused, influenced, participated_in, located_at, related_to,
before, after, contemporary_with, part_of, ruled, traded_with,
invented, discovered, practiced, spoke                                      # 15 个
```

### 3.2 冻结决策

**保留全部 15 个，additive 新增 3 个：`inherited` / `conquered` / `spread`。冻结后共 18 个。**

| 关系 | 状态 | 方向语义 | 消费者价值（为什么） |
|---|---|---|---|
| inherited | 新增 | A `inherited` B（A 继承 B） | JT-2："罗马继承了希腊的艺术/法律"——叙事主线 |
| conquered | 新增 | A `conquered` B（A 征服 B） | JT-2：征服是文明间最强、最易理解的因果链 |
| spread | 新增 | A `spread` B（A 使 B 传播） | JT-2：技术/信仰/作物的传播路径，跨文明探索的灵魂 |

**为什么是这 3 个，不是别的**：
- 它们共同覆盖"传承 / 征服 / 传播"三种**消费者最能感知的跨文明叙事动线**，直接服务 JT-2。
- 用户设计稿里的 `contemporary` **不新增**——它已存在为 `contemporary_with`（schema v2 §3），新增会造成同义冗余、破坏契约。
- 不新增 `Place`/`Location` 类关系——位置已用 `located_at` + `part_of` 表达。

### 3.3 命名冲突规避（重申 M3.5 Review §2）

- `contemporary` → 沿用 `contemporary_with`（**不新增**）。
- 任何新关系都不与现有 15 个同义。
- 关系元数据键 `source`/`target`（边端点）与 `citation`（出处）严格区分（schema §3 规则），M3.5 不改动。

### 3.4 向后兼容

- Additive：只增枚举，不改已有 15 个的语义或形状。
- `weight` / `confidence` / `valid_time` 仍为可选、可全 null（schema §3 已预留）——**v1 不依赖它们填充**。

### 3.5 冻结清单（最终）

```

before, after, contemporary_with, part_of, ruled, traded_with,
invented, discovered, practiced, spoke,
```

## 4. 冻结：GlobalGraph 数据规则

> GlobalGraph 是 Knowledge Layer 内的一个新模块（`core/global_graph.py`），**不是独立 Service**。它复用 `KnowledgeService` 已建的注册表（含 global_id 索引），只负责"把全量边（含跨主题）组成一张统一邻接图"。

### 4.1 数据来源（不新增存储）

- 来源 = `KnowledgeService` 已加载的全部 topic 数据集（经 `JsonTopicRepository`）。
- **不引入数据库 / 文件 / 缓存层**。GlobalGraph 是进程内、构建自既有的内存索引（同 `_ENTITY_INDEX` 接缝）。

### 4.2 global_id 规则（重申 schema §5）

- 格式 `namespace:id`，**只存在于数据，绝不进入 URL**（M2 路由正则 `^[a-z0-9_-]+$` 不变）。
- 在 GlobalGraph 中，`global_id` 是节点的**规范键（canonical key）**；`id` 仅用于同主题内查找。每个 `global_id` 对应唯一节点。

### 4.3 跨主题边纳入规则（本冻结的核心）

- **一条边若端点含 `:`（即 `namespace:id` 形式），它就是跨主题边，必须纳入 GlobalGraph。**
- 这与 per-topic `KnowledgeGraph`（`graph.py:161-166` 把此类边当 "dangling-by-design" 丢弃）**正好相反**——GlobalGraph 的存在意义就是把它们连起来。
- 跨主题边与同主题边在 GlobalGraph 中**同等对待**，统一以 `global_id` 为端点键。
- 边身份 = `(source_global_id, target_global_id, type)`；方向保留。

### 4.4 图索引与路径引擎边界

- **图索引**：启动时由 `KnowledgeService` 构建统一邻接表（out / in），与现有 `DirectedGraph` 算法（bfs / find_path / cycles）复用。
- **路径引擎**：确定性 BFS / 最短路径，作用于统一邻接。无随机性、无 ML。
- **不实现**：图数据库写入、分布式图计算、第三方图库。

### 4.5 关系权重处理（v1）

- `weight` / `confidence` / `valid_time` 全 null 时，引擎用静态 `RELATIONSHIP_MEANING` 语义权重表兜底（按关系类型，约 0.4–1.0），未知类型默认 0.4；仅当 `weight` 显式非空时才采用该显式值。
- v1 **不要求**任何数据填充这些字段；填充后引擎自动采用（additive 消费）。

### 4.6 Neo4j 预留接缝

- GlobalGraph 的所有查询经 `KnowledgeService` facade 暴露（如 `get_global_graph()` / `find_global_path()`）。
- 未来切 Neo4j：仅替换 `core/global_graph.py` 的内部实现（或新增 `Neo4jGlobalGraph`）并改组合根一处（`main.py:73`），**公共 API 形状不变**。
- 本冻结明确：M3.5 **不实现** Neo4j，仅保证接缝存在。

### 4.7 红线（冻结不可越）

- 无 AI / LLM、无 GIS / 地图、无推荐算法、无新增第三方依赖。

## 5. 冻结：Exploration Engine 产品定位

### 5.1 定义

**Exploration Engine = 给定入口实体，确定性地产出"排好序的探索路径与关联"，帮助消费者理解跨文明、跨时间的连接，且每条结果都可解释。**

### 5.2 v1 能力边界（四类消费者输出）

| 输出 | 对应 JT | 说明 |
|---|---|---|
| **Related（即时邻居，已排序）** | JT-1 | 在 M3.5-003 补全跨主题 `other`；排序用 §5.3 规则 |
| **Paths（多跳发现）** | JT-3 | "从罗马到丝绸之路，经由 X"——确定性 BFS/最短路径 |
| **Themes（按主题浏览）** | JT-2 | 按实体类型 / 关系类型筛选（如只看 `spread` 传播链） |
| **Timeline（时间感知）** | JT-1/2 | 结果按 `TimeValue` 排序，呈现先后与共存 |

### 5.3 确定性算法如何服务叙事（v1，详细在 M3.5-002）

- **路径发现**：BFS / 最短路径，跨主题邻接上运行。
- **四维评分排序**：实体类型权重 × 时间邻近度 × 关系方向 × 关系类型权重。
- **时间因素**：用 `TimeValue.value`（BCE 为负）做先后/共存判断。
- **权重兜底**：`weight` 为 null 时按关系类型采用 `RELATIONSHIP_MEANING` 静态语义权重（约 0.4–1.0），未知类型默认 0.4。
- **可解释性（JT-4）**：每条推荐附带"为什么"——关系类型 + 路径，前端可展示。

### 5.4 不在 v1 范围（明确划掉，防止范围蔓延）

- 无 AI 生成摘要、无个性化推荐、无 GIS 地图、无社交/协作、无编辑/写入。
